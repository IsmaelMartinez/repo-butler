# Dynamic Dashboard Research

Date: 2026-04-08

## Problem

The dashboards are static HTML generated once daily (2am UTC cron) or on manual trigger. Code changes to report templates require a forced regeneration to appear, and the data itself (issues, PRs, vulnerabilities) can go stale within hours. The report caching system (SHA-256 hash of snapshot) makes this worse — presentation-only changes don't invalidate the cache because the snapshot data hasn't changed.

The current delay chain: code change merged → manual workflow trigger → observe phase runs → cache check → report generation → Pages deploy. Even with force-report, this takes ~3 minutes. For data freshness, the daily cron means a new critical vulnerability won't appear on the dashboard for up to 24 hours.

## Current Architecture

Static HTML generated by Node.js string templates (`report-portfolio.js`, `report-repo.js`). Charts use Chart.js loaded from CDN. Data comes from GitHub API calls made during the OBSERVE phase and stored as JSON snapshots on the `repo-butler-data` orphan branch. Reports deploy to GitHub Pages as plain HTML files.

Constraints from CLAUDE.md: zero npm dependencies, zero infrastructure cost, GitHub Pages hosting, runs as a GitHub Action.

## Options

### Option 1: Client-side data fetching (hybrid static + dynamic)

Keep the static HTML shell but fetch live data for fast-changing metrics (open PRs, open issues, vulnerability counts) via the GitHub API directly from the browser. The static report provides the baseline, and a small inline `<script>` fetches fresh counts on page load.

Pros: no build system change, no new dependencies, instant data freshness for key metrics, works with current GitHub Pages hosting, maintains zero-dependency backend.

Cons: requires a GitHub token in the browser (public API has 60 req/hr limit for unauthenticated), CORS considerations, adds client-side complexity, rate limiting across 14 repos could exhaust anonymous quota quickly.

Mitigation: use the unauthenticated API for public repos (60 req/hr is enough for a single user viewing the dashboard), cache responses in sessionStorage, only fetch the 2-3 most volatile metrics (open issues count, open PRs count, latest vuln severity). For private repos this won't work without a token.

### Option 2: Increase pipeline frequency

Run the pipeline more often — every 4 hours instead of daily. The OBSERVE phase takes ~30 seconds, report generation ~2 minutes. GitHub Actions gives 2000 minutes/month on free tier.

Pros: no architecture change, simple cron update, data freshness improves from 24h to 4h.

Cons: doesn't solve the presentation cache problem (code changes still need force-report), 6 runs/day × 3 min = 18 min/day of Actions usage, still not real-time.

### Option 3: Astro (or similar SSG) with partial hydration

Replace string template HTML generation with Astro. Static pages at build time, with interactive islands for live data. Astro outputs plain HTML/CSS/JS — compatible with GitHub Pages. Could use Astro's content collections for snapshot data and component-based report sections.

Pros: component-based templates (easier to maintain than string concatenation), partial hydration for live data widgets, built-in asset optimization, better developer experience for iterating on dashboard design.

Cons: adds a build dependency (breaks the zero-dependency constraint), more complex CI pipeline, learning curve, Astro is Node-based so still generates static output for Pages. The zero-dependency constraint is a deliberate moat — adding Astro means adding hundreds of transitive dependencies.

### Option 4: Fix the cache invalidation problem only

The immediate pain point isn't data freshness (daily is acceptable for most metrics) — it's that presentation changes don't take effect without force-report. Fix this by including a hash of the report template files in the cache key alongside the snapshot hash. When report code changes, the cache automatically invalidates.

Pros: minimal change (a few lines in report.js), solves the immediate problem, no architecture change.

Cons: doesn't improve data freshness at all, only fixes the deployment delay for code changes.

### Option 5: Webhook-driven incremental updates

Use GitHub webhooks (via a lightweight serverless function) to trigger report regeneration when specific events occur: issue opened/closed, PR opened/merged, security advisory published. Only regenerate the affected repo's page, not the full portfolio.

Pros: near-real-time updates for the metrics that matter most, efficient (only regenerates what changed).

Cons: requires infrastructure beyond GitHub Actions (a webhook receiver — AWS Lambda, Cloudflare Worker, or similar), adds operational complexity, breaks the zero-infrastructure constraint.

### Option 6: Astro component for ismaelmartinez.me.uk integration

Instead of treating Astro as a standalone migration, build the dashboard as an Astro component that integrates into the existing personal website (ismaelmartinez.me.uk). The repo-butler pipeline continues to run daily and publish JSON snapshot data to the `repo-butler-data` branch. The Astro site consumes this JSON at build time for the static baseline, and uses client-side islands for live data.

Architecture: repo-butler stays as-is (zero-dependency GitHub Action producing JSON snapshots). A new `astro-dashboard` package (or directory in the personal site repo) provides Astro components that render the portfolio and per-repo views. The components fetch snapshot JSON from the `repo-butler-data` branch at build time (via GitHub raw content URL or API), and hydrate interactive islands that fetch live metrics (open PRs, issues, vulns) from the GitHub API on page load.

This separates concerns cleanly: repo-butler observes and stores data, the Astro site presents it. The current string-template HTML reports remain as a standalone fallback (or are retired once the Astro integration is stable).

Pros: integrates with the personal website's existing Astro build pipeline, partial hydration gives live data for volatile metrics without full client-side rendering, component-based templates are easier to iterate on than string concatenation, the dashboard gets the site's existing design system and navigation, repo-butler stays zero-dependency (it just produces JSON).

Cons: the dashboard lives in a different repo (ismaelmartinez.me.uk), so changes require coordinating two repos. Build time increases for the personal site. The Astro components need to understand the snapshot schema.

Data flow:
```
repo-butler (GitHub Action, daily)
  → OBSERVE → JSON snapshot → repo-butler-data branch
  → REPORT  → static HTML   → GitHub Pages (fallback)

ismaelmartinez.me.uk (Astro, on deploy)
  → fetch snapshot JSON from repo-butler-data branch at build time
  → render static baseline with Astro components
  → hydrate islands: fetch live PRs/issues/vulns from GitHub API on page load
```

Live data islands would cover: open PR count per repo (unauthenticated `GET /repos/{owner}/{repo}/pulls?state=open`), open issue count (already in snapshot but refreshable), latest vulnerability severity (requires auth — stays static from snapshot).

## Recommendation

Option 6 (Astro component for personal site) is the right direction given the integration goal. It keeps repo-butler focused on observation and data collection (its strength), moves presentation into the Astro site where it benefits from the existing design system and build pipeline, and enables partial hydration for live metrics.

The implementation path is: first, ship Option 4 (cache invalidation fix) as a quick win for the current dashboards. Then build an Astro component that consumes the snapshot JSON and renders the portfolio view. Start with a static-only component that matches the current dashboard layout, then add hydrated islands for open PRs and issues. The current GitHub Pages reports stay as a fallback until the Astro integration is stable.

The key architectural decision is where the Astro components live — in the personal site repo (simpler deployment) or as a publishable package in the repo-butler repo (more portable). Given that there's only one consumer, putting them directly in the personal site repo is simpler.

## Performance: speeding up the pipeline

The current pipeline takes ~3 minutes end-to-end: OBSERVE (~30s), portfolio details fetching (~60s, dominated by API calls), libyear computation (~30s, sequential npm registry lookups), report generation (~10s), and Pages deploy (~60s). The bottleneck is overwhelmingly I/O (GitHub API calls and npm registry lookups), not CPU. Changing the language won't help if you're waiting on HTTP responses.

### Where the time actually goes

The OBSERVE phase makes ~13 API calls in parallel via `Promise.all` — this is already well-optimised. The `fetchPortfolioDetails` function is the heaviest: 12 parallel API calls per repo × 14 repos = ~168 API calls, but they're batched per-repo with `Promise.all` inside a `.map()` that runs all 14 repos concurrently. The serial bottleneck is libyear computation, which runs one repo at a time to avoid fanning out npm registry requests.

Report generation (building HTML strings) takes negligible time — under 100ms for all 14 repos. This is the part a faster language would speed up, but it's already instant.

### Option A: Parallelise libyear computation

The libyear loop is sequential (`for...of` with `await`). Batching it into groups of 3-4 concurrent repos would cut libyear time by 3-4x without overwhelming the npm registry. This is a ~5-line change.

### Option B: Incremental report generation

Only regenerate reports for repos whose snapshot data actually changed. The current cache is all-or-nothing (one hash for the entire portfolio). Per-repo hashing would let unchanged repos skip regeneration entirely. For a typical daily run where only 2-3 repos had activity, this skips ~80% of the API calls in `fetchPortfolioDetails`.

This is the biggest potential win: instead of 168 API calls, you'd make ~36 (for 3 active repos) plus a lightweight check for the rest.

### Option C: Rust/Go rewrite for the pipeline

Rewriting the OBSERVE and REPORT phases in Rust or Go would give faster startup (~5ms vs ~50ms for Node), lower memory usage, and truly parallel HTTP via async runtime (tokio/goroutines). But the actual gains would be minimal because the time is spent waiting on GitHub API responses, not processing data. A Rust version waiting 200ms for an API response is no faster than a Node version waiting 200ms.

Where Rust/Go would genuinely help: if the pipeline processed hundreds of repos, the concurrent HTTP handling and lower per-request overhead would scale better. For 14 repos, it's not the bottleneck.

If pursuing this path, Go would be the better choice — simpler concurrency model (`go func()` vs tokio), single binary distribution (no runtime needed in the Action), and the team already has Go experience (github-issue-triage-bot is Go). The Action would use `using: 'composite'` with a pre-built binary instead of `using: 'node20'`.

### Option D: Pre-compiled Action with ncc or esbuild

Bundle the Node.js code into a single file with esbuild (zero-config, fast). This eliminates module resolution overhead at startup. Combined with the `node20` runtime, startup drops from ~50ms to ~20ms. Marginal gain but zero effort.

### Recommendation

The biggest wins for pipeline speed, in order of impact:

1. Incremental report generation (Option B) — skip unchanged repos, cut API calls by ~80% on typical runs. Medium effort, high impact.
2. Parallelise libyear (Option A) — 5-line change, ~30s saving. Low effort, moderate impact.
3. Fix cache invalidation (Option 4 from above) — ensures code changes deploy without force-report. Low effort, fixes the UX problem.

A language rewrite (Option C) is not justified for 14 repos where I/O dominates. If the portfolio grows to 50+ repos, revisit Go as the execution layer — but by then the incremental approach (Option B) would matter even more.
