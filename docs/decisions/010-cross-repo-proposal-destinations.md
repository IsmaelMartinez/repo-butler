# ADR-010: Cross-Repo Proposal Destinations

Date: 2026-06-22
Status: Proposed

Amended: 2026-06-22 — the three open questions the Boundary preservation section below leaves to the implementation are settled here and in [ADR-011](011-portfolio-informed-generic-proposals.md) (that section is left unedited as the original record of them being open): a target repo MUST be onboarded (carry the `repo-butler` marker `onboard.js` adds to `CLAUDE.md`) before it can receive any cross-repo issue; each cross-repo issue back-links to a single host-side tracking issue via a bare allowlisted URL (never a `#`-cross-reference, which would auto-notify a third repo); and a governance idea naming a target not on `propose-targets` falls back to the host backlog while the allow-list is empty. Separately, this ADR scoped only the closed catalogue of governance findings; the broader portfolio-informed-generic idea class (small generic improvements justified by a cross-repo statistic) is a new decision recorded in [ADR-011](011-portfolio-informed-generic-proposals.md), which reuses this ADR's `targetRepo` field and five-gate adaptation verbatim and adds two tightenings (a two-axis volume cap and cross-reference autolink neutralisation). For the generic-class expansion, ADR-010 is superseded-in-part by ADR-011.

## Context

The butler already ideates across the whole portfolio. When governance findings exist — and the daily pipeline refreshes them four times a day — `buildIdeatePrompt` (`src/ideate.js:58`) switches from a single-project advisor persona to a portfolio governance advisor, feeds the LLM the full cross-repo findings via `appendGovernanceContext`, and instructs it to reference specific repos and cross-repo statistics ("configured in 14/19 repos") and to justify each idea as a portfolio-level concern. The cross-repo visibility that is the butler's defining capability (ADR-002) is therefore already used to generate proposals.

What is not portfolio-wide is where those proposals land. PROPOSE reads `owner`/`repo` off the shared context — which is always the host repo, `IsmaelMartinez/repo-butler` — and posts every issue to `/repos/${owner}/${repo}/issues` (`src/propose.js:177`, `:266`). A proposal whose entire rationale is "repo-X is missing Dependabot and 14 other repos have it" becomes an issue in the butler's own tracker, not in repo-X. The reasoning is cross-repo; the destination is single-repo. That mismatch is why the value of the portfolio view feels under-realised: the butler does the cross-repo thinking and then files all of it into one backlog.

This sits between two existing tracks that deliberately do not cover it. The first is the Governance Apply path (ADR-005, ADR-007 Track A): `src/apply.js` already writes across repos, opening templated config PRs (`dependabot.yml`, `CODEOWNERS`, `SECURITY.md`, code-scanning workflows) behind five layered gates. But that path is restricted by construction to deterministic, static-file remediations — it never writes free-form LLM content into another repo. The second is ADR-007 Track B, the agent-driven path that will open code-change PRs for findings a template cannot encode; that is about writing code, not filing proposals, and remains design-only. The user-facing gap — surfacing the butler's governance proposals as issues in the repos they concern — is covered by neither, and is what this ADR addresses.

Two boundaries from prior decisions constrain the shape of any answer. ADR-002 cedes generic per-repo improvement ideas ("fix this bug, refactor this module in repo-X") to the triage bot, which has the per-repo context the butler lacks; the butler's lane is cross-repo governance only. And `ROADMAP.md:148` already records that PROPOSE-on-a-schedule is out of scope until IDEATE has produced trustworthy council-approved proposals for at least a month, because PROPOSE creates real issues and carries spam-risk blast radius. Cross-repo PROPOSE inherits both constraints: it must stay within the governance lane, and it raises the same blast-radius question the scheduling note already flags, multiplied across every target repo rather than concentrated in one.

## Decision

Add a target-repo destination to PROPOSE for governance-class proposals only, gated by an adaptation of the ADR-005 five-gate model, shipped default-closed, and graduated on the same falsifiable track record discipline that ADR-007 Track A used. This is a destination relaxation of an existing phase, not a new phase and not a rewrite of IDEATE.

### What changes

Each governance-class idea carries the target repo it concerns. Governance findings already name their repos — `f.repo` on policy-drift, tier-uplift, and stale-Dependabot findings, and `f.nonCompliant` (an array of repo names) on standards gaps — so the structured idea spec gains a `targetRepo` field that IDEATE populates from the finding and that the existing field-loop parser reads alongside `TITLE`, `PRIORITY`, and the rest. PROPOSE then files each such idea against its target repo rather than the host. The mechanics are small because the per-repo machinery already generalises: `findDuplicates` and `findDuplicatePRs` already take an `(owner, repo)` pair, so duplicate detection works unchanged against the target repo, and `validateIssueBody` already runs on the fully composed body before any write. A proposal with no target repo, or a non-governance idea, continues to file into the host backlog exactly as today.

### Why the trust delta is real but bounded

Writing LLM-authored, free-form issue bodies into other repos is a larger trust delta than the templated-config writes APPLY performs, and it must not borrow APPLY's gates by assumption. The risk is two-sided. An issue is non-destructive — it changes no code and no settings, and closing it is trivial — so the per-write stakes are lower than a PR or a merge. But the reach is wider: a bug or a prompt-injection-shaped proposal could file noise into every target repo's tracker and notify every watcher at once, which is precisely the blast radius ADR-005 was written to fence. The output-side defence already exists (`validateIssueBody` blocks @mentions, keys, and disallowed URLs; `sanitizeForPrompt` strips injection patterns from finding data before it reaches the LLM; `REPO_NAME_PATTERN` validates any repo name interpolated into a write), but the trust model was authored assuming templated content, so cross-repo PROPOSE is treated as a new class that earns its own gates rather than inheriting them.

### Gates

The five gates adapt as follows. `require_approval` keeps PROPOSE's existing semantics, which are the inverse of ADR-005's — and the ADR must not conflate the two. In PROPOSE, `require_approval` defaults to true and that *blocks* real issue creation: the phase logs what it would file and writes nothing (`src/propose.js:196-206`); only setting it to false lets PROPOSE create issues at all. (On the apply side, ADR-005's `require_approval=true` is the opposite — the master switch that must be true for apply to run.) Cross-repo PROPOSE inherits PROPOSE's flag unchanged, so it is default-closed on two independent axes: nothing is filed anywhere, host or cross-repo, until the operator sets `require_approval: false`, and nothing goes cross-repo until the `propose-targets` allow-list is also non-empty. Dry-run stays fail-closed: anything other than the literal `'false'` logs what would be filed and writes nothing. The allow-list (working name `propose-targets`, default empty) controls which target repos are eligible, mirroring how `apply-schedule` and `apply-automerge` keep cross-repo writes default-closed and per-class; with it empty, the host backlog is the only destination and behaviour is byte-identical to today. The per-run cap is retained and tightened to bound both total issues and issues-per-repo. Each idea's `targetRepo` is validated against `REPO_NAME_PATTERN` and checked for `propose-targets` membership *before* any repo-specific API call — `findDuplicates`, `findDuplicatePRs`, `ensureLabel`, the issue POST — following the gate-before-write ordering `apply.js` already uses, so a hallucinated or injection-shaped repo name never reaches an arbitrary API path. The existing duplicate-and-recently-declined-PR dedup then guards against re-proposal churn.

### Boundary preservation

Only proposals that originate from cross-repo findings go cross-repo. Generic per-repo ideas remain ceded to the triage bot per ADR-002, so the butler does not start filing "refactor this module" issues into repos where it lacks the context to justify them. The council still gates which proposals survive to PROPOSE at all, so the cross-repo channel carries only council-approved governance proposals. Several questions are left to the implementation. Whether a target repo should be required to be onboarded (carry the `CLAUDE.md` marker `onboard.js` adds) before it can receive proposals. Whether each target-repo issue should back-link to a portfolio tracking issue in the host repo so the maintainer retains one place to see everything the butler has proposed. And what becomes of a governance idea that names a target repo not on `propose-targets`: filing it to the host backlog (today's behaviour) keeps it visible but may reaccumulate the clutter this feature aims to reduce, so the implementation should choose between host-backlog fallback, a silent skip, or a configurable option — with host-backlog fallback the natural default while the allow-list is empty, since that is the byte-identical-to-today case.

### Staging

This is deferred behind the same unblock condition the roadmap already names for PROPOSE graduation: IDEATE producing trustworthy, council-approved governance proposals for at least a month, with the host-backlog proposals reviewed and acted on rather than ignored. It ships default-closed (empty `propose-targets`), dry-run first, and graduates to live only in a separate reviewed change once the soak is clean — the two-axis, separately-graduated pattern ADR-007 Track A established. It is the natural complement to the scheduling graduation at `ROADMAP.md:148`: scheduling decides when PROPOSE runs, this decides where it writes, and both are gated on the same track record before either goes live.

## Consequences

The butler's portfolio view stops being a closed loop. Governance proposals surface in the repos they concern, where the relevant maintainer or watcher will see them, instead of accumulating in a single backlog that only the portfolio owner reads. This makes the cross-repo capability legible to the rest of the portfolio without expanding the butler's scope beyond governance.

The cost is a new cross-repo write surface that carries free-form content, which is a genuine trust step beyond the templated-config writes shipped to date. The mitigation is to inherit the discipline rather than the gates: default-closed, dry-run-first, allow-listed per target, capped, and graduated only on a clean track record. The never-auto-merge rule is untouched and irrelevant here — issues are not merged — but the spirit of ADR-005, that no single failure can blast across the portfolio unbounded, is preserved by the cap and the allow-list.

The implementation is left to a separate reviewed PR, which must settle the open questions above (the exact config surface, the onboarding precondition, and whether to thread a host-side tracking issue) and add the `targetRepo` field to the idea spec, the parser, and the IDEATE prompt. This ADR records the design and the gates so they are agreed before any code is written, consistent with how ADR-005 and ADR-007 required their cross-repo trust changes to be settled in advance.
