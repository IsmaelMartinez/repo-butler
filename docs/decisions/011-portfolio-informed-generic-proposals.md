# ADR-011: Portfolio-Informed Generic Proposals

Date: 2026-06-22
Status: Proposed

Amended: 2026-07-14 (finalised 2026-07-20) — the Graduation section below names tier-uplift as the safest first class to flip live from the inherited governance catalogue, on the strength of its deterministic body. That reasoning predates G9 (#304), which made every cross-repo class's body deterministic and so neutralised the differential; and the completed month-long G10 soak anchored its sustained evidence to standards-gap (gate-admitted candidates in two separate runs: 2026-07-13 and 2026-07-20), while tier-uplift's first admitted candidate arrived only on the final run (2026-07-20, ai-model-advisor). The first graduation therefore flips standards-gap on that track record; tier-uplift now has the start of its own evidence base and follows later in its own reviewed change, per the one-class-at-a-time discipline — the full gate analysis and the allow-list scope are recorded in ADR-010's "G10 graduation" note. The section below is left unedited as the original record. This amendment concerns only the inherited governance catalogue; the net-new generic classes (G11: description-gap, topics-gap) still require their own re-gathered soak before any of them graduates, exactly as the Decision requires.

## Context

ADR-010 (Status: Proposed, design-only) proposed adding a `targetRepo` destination to PROPOSE so the butler could file an issue into the repo a proposal concerns rather than its own backlog, but it scoped that strictly to the closed catalogue of governance findings — standards gaps, policy drift, tier uplift, and stale-Dependabot audits — each of which already names its repo (`nonCompliant` on standards gaps at `src/governance.js:167`; `f.repo` on policy-drift, tier-uplift, and stale-Dependabot findings). The maintainer now wants something one step broader: the butler should be able to propose small, generic improvements to other repos — the kind of nudge that is not a named standard or compliance gap but is still worth raising — justified by the overall portfolio observability the butler alone has.

That expansion crosses two prior decisions and needs them settled in writing before any code, the same discipline ADR-005 and ADR-007 imposed on every cross-repo trust change. It crosses ADR-002, which deliberately ceded generic per-repo improvement ideas ("fix this bug, refactor this module") to the triage bot because the butler lacks the per-repo code and issue context the triage bot's vector store and webhook pipeline provide. And it widens the free-form cross-repo write surface ADR-010 opened: where ADR-010 routes a bounded set of deterministic findings, a generic idea class admits a far larger population of LLM-authored proposals, so the gates that were comfortable for the governance catalogue become genuinely load-bearing.

The reframe that makes this coherent rather than a scope creep is that ADR-002's boundary was always really about the source of justification, not the subject. ADR-002 grounds the butler's lane in the one capability nothing else in the ecosystem has — the cross-repo view — and a generic-subject nudge that is justified by a cross-repo statistic the butler computes and the per-repo triage bot structurally cannot is squarely inside that lane. "Most of your active repos set a repository description; this one does not" is a comparison the butler measures across the portfolio, not a fact that required reading the repo's code. This ADR records the boundary refinement, the trust model for the wider class, and what stays out.

## Decision

Permit a new idea class — portfolio-informed-generic — whose subject may be a small generic improvement but whose admissibility is gated on the source of its justification. It reuses ADR-010's `targetRepo` field, parser, dedup, and five-gate adaptation unchanged, adds two tightenings the wider population requires, and ships default-closed and graduated exactly as ADR-010 staged the governance class. The accompanying ADR-002 refinement and ADR-010 amendment land alongside this ADR.

### The boundary is the source of justification, not the subject

A proposal is admissible for cross-repo routing only when its rationale rests on a cross-repo statistic the butler computes across the portfolio — an adoption fraction ("11 of 14 active repos do X"), a median or percentile rank, or a below-median drift magnitude — and never on a claim that required reading the target repo's code, tests, or issue contents. Per-repo code and content claims ("this function is buggy", "refactor this module", "this test is flaky", "rewrite this confusingly worded section") remain ceded to the triage bot, which has the embeddings and webhook context to ground them and which the butler does not. The butler may only assert what its cross-portfolio numbers support.

This is not a prompt convention; it is a hard, deterministic gate. The single enforcement that survives a model slip is to route an idea cross-repo only when its `targetRepo` derives from a deterministic governance or drift finding that already names that repo — preserving ADR-010's "only proposals that originate from cross-repo findings go cross-repo" discipline — and only when the idea's rationale matches a cross-repo-statistic pattern and matches none of the per-repo-code patterns. The eligibility filter that governance already applies (`eligibleRepos` at `src/governance.js:89`: not archived, not a fork, not excluded by `REPO_EXCLUSION_PATTERNS`) is honoured on the target, so an archived-but-previously-onboarded repo never receives a nudge.

### The council is a quality filter, never the safety boundary

It is tempting to let the agent council enforce the boundary, since `reviewProposals` already gates which ideas reach PROPOSE. It cannot. The council is itself an LLM inside the trust boundary — its default quick mode is a single call (`buildQuickDeliberationPrompt` at `src/council.js:128`) — and its deliberation never even sees `targetRepo`. Strengthening the deliberation prompt with a statistic-grounding rule and adding a confidence and priority gate for cross-repo-targeted ideas (`council_confidence` at `src/council.js:459`) raises the quality of the wider, riskier population and is worth doing, but it is defence-in-depth layered on top of the deterministic gate above, not a substitute for it.

### Trust model: ADR-010's five gates verbatim, plus two tightenings

The five-gate adaptation ADR-010 settled carries over unchanged, including the point ADR-010 was careful to make: PROPOSE's `require_approval` is the inverse of APPLY's. In PROPOSE, `require_approval` defaults to true and that blocks issue creation, forcing dry-run (`src/propose.js:196-206`); only false lets it write. It must never be conflated with APPLY's master-on switch (ADR-005). Keeping APPLY's templated-config trust model and PROPOSE's free-form trust model in separate ADRs is deliberate, to prevent exactly that conflation.

Two tightenings are required for the wider population, both surfaced by adversarial review of the design. First, a two-axis volume cap. PROPOSE today bounds only the total issues per run via a single slice; that bounds nothing per target repo, so one buggy or injection-shaped finding naming many repos could spread the budget across many trackers, and the obvious operator fix — raising the per-run cap — recreates the unbounded fan-out ADR-005 fences. The cap must bound both the per-run total (the portfolio-wide ceiling) and the issues per target repo (kept low, default one). Second, cross-reference autolink neutralisation. The existing output validators block the `@handle` mention form, but GitHub's `owner/repo#N` and bare `#N` cross-reference autolinks notify a third repo's participants with no `@` and pass every current validator; a composed body destined for another repo must reject those tokens, and the host-side tracking back-link (below) must therefore be a bare allowlisted URL, never a `#`-cross-reference. `validateIssueBody` already runs on the fully composed body and blocks keys, non-allowlisted @mentions, disallowed URL hosts, and over-length content destination-agnostically; that coverage is confirmed to extend to cross-repo bodies with no relaxation.

Implementation note (G3, shipped ahead of routing): cross-reference neutralisation lands as a deterministic `validateCrossRefs` helper in `safety.js` that rejects all three GitHub autolink forms — `owner/repo#N`, a bare `#N`, and the `GH-N` shorthand — invoked by `validateIssueBody` only when its new `{ crossRepo: true }` option is set. It is opt-in rather than applied to the default path because host bodies legitimately cite this repo's own issues as bare `#N` — the IDEATE prompt encourages it and the roadmap (validated through the same function in `update.js`) is full of them; the routing in G5/G9 passes `crossRepo: true` for cross-repo-destined bodies. Two residuals are recorded rather than silently assumed safe. First, `validateUrls` is host-scoped, not path-scoped: any `github.com` path passes, so the allowlist constrains the host of a link but not its path — acceptable here because the only cross-repo URL the butler emits is its own deterministic host-side tracking back-link, but a future change that lets the LLM emit arbitrary `github.com/...` paths would need path scoping. Second, `CORE_URL_HOSTS` is hardcoded to this portfolio's hosts, so an external `@v1` consumer whose tracking issue lives on its own Pages or enterprise domain would have that back-link dropped by `validateUrls`; making the allowlist configurable is deferred until an external consumer actually adopts cross-repo PROPOSE.

Implementation note (G4, the gate ahead of routing): the finding-anchoring boundary lands as a pure `resolveCrossRepoDestination(idea, { findings, eligibleRepoNames, owner })` in `safety.js`, returning `host`, `cross-repo`, or `drop` with a stable machine-readable reason code for the dry-run soak log. It enforces the boundary as a conjunction. A malformed `targetRepo` — anything outside `REPO_NAME_PATTERN` — is dropped outright rather than filed, because a name carrying slashes or shell metacharacters is injection-shaped and not a benign typo to fall back from. Otherwise the target must be named by a deterministic finding of a statistic-bearing class — standards-gap, policy-drift, or tier-uplift — with stale-Dependabot deliberately excluded as an anchor, since its justification is a temporal per-repo fact rather than the cross-portfolio comparison this class requires. The target must also be eligible, checked against an eligible-name set the caller supplies and fail-closed when that set is empty, which is defence-in-depth on top of the anchor. Finally the rationale must carry an actual quantitative cross-portfolio token — an adoption fraction, an N-of-M count, a percentage, or a median/percentile rank — while matching none of the per-repo-code patterns. An adversarial review of the first cut caught the statistic check admitting on bare topic vocabulary alone (a lone "portfolio" or "adopt" with no number), which would have let an off-topic nudge pass the only content gate; the gate now requires a quantity, with the digit count capped to plausible repo counts so a date or version slash pair is not mistaken for an adoption fraction. The function takes no dependency on `governance.js` — the caller supplies both the findings and the eligible-name set — so `safety.js` stays the single self-contained home for cross-repo write gates. Two residuals are recorded. The per-repo-code check is a keyword denylist, so the deterministic-body composition (G9) remains the real guarantee that a published cross-repo body asserts only the finding's actual statistic rather than the model's free text; and `owner` is echoed from context unvalidated by design, because it is always the host owner and never LLM-supplied. Wiring the gate into the PROPOSE write path, behind the `propose-targets` and `propose-classes` config, is G5.

### Config surface

Three default-empty entries join the existing `apply-cap`, `apply-schedule`, and `apply-automerge` blocks in `src/config.js`, expressed as the same key-presence maps the hand-rolled YAML parser accepts (flat plus one level of nesting, no list syntax): `propose-targets` (eligible target repo short-names), `propose-classes` (which finding classes route cross-repo, with a reserved key for the net-new generic class), and a per-target cap scalar under `limits`. `propose-classes` is the load-bearing graduation control and an addition beyond ADR-010's literal text, recorded here; it lets each class graduate independently exactly as `apply-schedule` promotes one tool at a time. The repository owner always comes from context and is never LLM-supplied; only the repo short-name varies, and it is validated against `REPO_NAME_PATTERN` and checked for `propose-targets` membership before any repo-specific API call. With every map empty — the default — behaviour is byte-identical to today.

### Destination, format, and onboarding precondition

The destination is always an issue; generic improvements are not in `TEMPLATABLE_TOOLS` (`src/governance.js:362`) and so never take the APPLY templated-PR path. Cross-repo issues carry a distinct `portfolio-nudge` label (in addition to the existing proposal and agent labels) so a target maintainer can see at a glance that the issue is bot-authored from cross-repo norms; the label is created on the target repo only after the gates pass. For the deterministic classes the body is composed from the repo name and the real finding statistic rather than free-form prose, and host-specific fields are suppressed so the model never asserts a target-repo fact it never saw — IDEATE runs once over the host snapshot, so an unsuppressed body would otherwise leak the host's affected-files and host issue references into another repo.

This ADR resolves ADR-010's three open questions. A target repo must be onboarded — carrying the `repo-butler` marker that `onboard.js` adds to `CLAUDE.md` — before it can receive any cross-repo issue; the onboarding PR text already announces that the butler may open governance proposals, so the marker is both a consent signal and a second blast-radius fence. Each cross-repo issue back-links to a single host-side tracking issue via a bare allowlisted URL, restoring the maintainer's one-place overview that routing would otherwise lose. And a governance idea naming a target not on `propose-targets` falls back to the host backlog while the allow-list is empty, which is the byte-identical-to-today case.

Implementation note (G9, the cross-repo issue format): the four format elements land in `propose.js`. A cross-repo body is composed DETERMINISTICALLY by `buildCrossRepoIssueBody(finding, { trackingUrl })` — which takes the finding and tracking URL but never the idea, so the no-leak property is structural — from the anchoring finding's own statistic — adoption fraction for a standards-gap, the expected/actual comparison for a policy-drift, the current/target tier and named failing checks for a tier-uplift — and never from the LLM's `rationale`, `currentState`, `affectedFiles`, or host issue references, so the host snapshot cannot leak into another repo. This is the real guarantee the G4 note deferred here: the published body asserts only what the butler computed, which the G4 per-repo-code denylist could only approximate. The anchoring finding is re-found in `propose()` via the now-exported `findingNamesRepo` and the gate's `anchorType`, leaving the security-critical gate return shape untouched. The body still passes `validateIssueBody({ crossRepo: true })` as defence-in-depth, which holds trivially because the deterministic text carries only bare allowlisted URLs. The distinct `portfolio-nudge` label is added to the target's label set (and created on the target only after the gates pass) and never to the host's. The onboarding precondition is a per-run-cached `hasOnboardingMarker` read that fails closed — a missing `CLAUDE.md`, an absent marker, or any API error routes the idea to the host backlog (`reason: target-not-onboarded`) rather than crossing — checked before the per-target cap so a downgraded idea is never wrongly capped. The single host-side umbrella tracking issue is found-or-created by a stable title via `ensureTrackingIssue` (never written in dry-run, null on any error so it never blocks a nudge), and each cross-repo body links back to it with a bare `github.com` URL — never a `#`-cross-reference, which `validateCrossRefs` (G3) would reject. Two residuals are recorded. The issue title is still LLM-authored (only the body is deterministic per this ADR), but a pre-PR review surfaced that an unvalidated title reaches a foreign repo unseen by the body validators, so `validateIssueTitle` gained a `crossRepo` mode that holds the title to the same content gates as a cross-repo body — cross-reference autolinks, `@`-mentions, disallowed URLs, and per-repo-code/content claims — and `propose()` skips any cross-repo idea whose title fails it; making the title fully deterministic remains a possible future tightening but is no longer a gap because the title now clears the same boundary the body does. And the umbrella issue's body is static butler-authored text, so it is filed without `validateIssueBody`, consistent with how `onboard.js` files its static PR body.

### Default-closed, and external consumers

The capability is default-closed on three independent axes — the two config-level gates ADR-010 names plus the env-level dry-run gate it folds into the same fail-closed discussion: `require_approval` defaults to true and blocks writes, `INPUT_DRY_RUN` must be the literal string `false` to write, and `propose-targets` must be non-empty for any cross-repo destination. A fresh external consumer using `uses: IsmaelMartinez/repo-butler@v1` with no `roadmap.yml` inherits all three defaults and so receives nothing until it deliberately opts in. On that basis the decision is to ship this on `@v1` rather than gate it behind a `@v2` major bump: it activates for no one without an explicit, reviewed config change, exactly as every prior cross-repo capability shipped.

### Graduation

The graduation bar is ADR-010's, with one sharpening the adversarial review insisted on: the soak evidence must be re-gathered for the wider portfolio-informed-generic class, because the governance-only track record does not cover it. The class soaks in dry-run for at least a month while the extended dry-run log records every intended destination, then a single safest class graduates live in a separate reviewed PR that stops for explicit human merge — never autonomous, never chained "if green, merge". Tier-uplift is the safest first class to graduate live from the inherited governance catalogue: its body is a deterministic list of named failing checks and it already names its repo. (That is distinct from the first net-new generic classes — description-gap and topics-gap — discussed below; "first" here means the first class flipped live, not the first new subject.) Each subsequent class is promoted by adding one `propose-classes` key in its own reviewed PR and is reversible by removing it. Where the butler writes stays independent of when it runs: putting PROPOSE on a schedule is a separate axis ADR-010 and the roadmap already track, and is out of scope here.

### What stays out

Deep per-repo code and content analysis stays ceded to the triage bot. Cross-repo PRs and file writes stay APPLY's lane (`TEMPLATABLE_TOOLS`). PROPOSE-on-a-schedule stays a separate decision. No list syntax enters the config, no npm dependency is added, and the dormant triage-bot integration is not revived.

For the net-new subject matter, the cleanest first net-new classes are description-gap and topics-gap, computed deterministically over `eligibleRepos` against the `MAJORITY_THRESHOLD` adoption fraction (`src/governance.js:75`) from the `description` and `topics` fields the portfolio path already carries (`src/observe.js:195`, `:207`); their bodies are fully deterministic, which narrows the free-form trust surface. The maintainer's headline example — CONTRIBUTING-quickstart adoption — and a pr-template-gap class are deferred, not shipped, because the portfolio detail object carries only aggregate community-health, not the per-file CONTRIBUTING and pull-request-template flags those classes would need; asserting that fraction today would be a hallucinated statistic no validator can catch, so those classes wait behind a small plumbing change to carry the per-file community-profile map. Explicitly excluded as triage-bot or non-portfolio-comparable: anything keyed on issue titles, labels, or reaction counts; per-repo bug counts; bus factor and time-to-close median (single-repo summary signals, not on the portfolio path); and libyear (ecosystem-specific).

## Consequences

The butler's generic-improvement value reaches the repos it concerns while the boundary and the trust model hold. The ADR-002 boundary is sharpened from subject to source-of-justification without changing its substance, so a future reader does not apply the old subject test and wrongly accept a code-claim proposal or reject a legitimate portfolio-informed one. APPLY's and PROPOSE's trust models stay in separate ADRs, avoiding the conflation ADR-010 warned against. The load-bearing safety is the deterministic finding-anchoring gate, with the council as a quality filter and the two-axis cap and onboarding precondition as blast-radius fences; the never-auto-merge rule is untouched because issues are not merged, and the spirit of ADR-005 — that no single failure blasts the portfolio unbounded — is preserved by the cap and the allow-list.

The implementation is deferred to a phased plan whose goals each land as their own reviewed PR, default-closed and dormant until a clean month-long soak of the new class, at which point a single class graduates live in a separate reviewed change. This ADR and its companion edits to ADR-002 and ADR-010 record the design and the gates so they are agreed before any code is written.
