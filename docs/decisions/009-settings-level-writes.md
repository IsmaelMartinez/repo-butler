# ADR-009: Settings-Level (PR-less) Cross-Repo Writes

Date: 2026-06-13
Status: Accepted

Amends [ADR-005](005-cross-repo-write-trust-model.md) (cross-repo write trust model) and extends [ADR-007](007-agents-and-execution.md) Track A (templatable findings). Accepted 2026-06-13 after a five-persona council review (Security, Stability, Maintainability, Development, Product); the hardening from that review is folded into the Decision below. The code ships dry-run-inert and manual-dispatch only; going live additionally requires a GitHub App permission grant (see Token scope) and is the maintainer's deliberate step.

Go-live (2026-06-15): the go-live gate described under Token scope has been satisfied. The maintainer granted the App `administration: write` and approved the installation, and the `copilot_code_review` ruleset is now live on all 13 portfolio repos (PR #271 detection + PR #272 auto-enable), bringing Copilot code review online ahead of the Gemini Code Assist sunset. The permission gate is therefore satisfied — live ruleset writes are now possible — so the "until then the feature is inert" / "until the grant is made" wording below is historical. This does not change the default behaviour: the settings apply still runs manual-dispatch only (`apply.yml`, `tools=code-review-bot`) and dry-run fail-closed by default, and is deliberately kept off the `apply-schedule` allow-list per ADR-007 (a settings write has no PR-review moment). A live write still requires a deliberate `dry-run=false` dispatch — the gate being open means such a run now succeeds rather than 403-ing, not that the workflow mutates settings on its own.

Scheduled promotion (2026-07-13): the separately-reviewed config change this ADR anticipated has landed — `code-review-bot` is now on the `apply-schedule` allow-list, so the weekly no-human scheduled run may enable the ruleset on a newly non-compliant repo. The track-record condition in the Decision below is met on both counts: the `codeowners`/`security-md` PR classes produced the nineteen-PR record that graduated ADR-007 stage 4 to live (2026-06-15), and the settings class earned its own — the 2026-06-15 live rollout enabled 13/13 repos with zero errors (independently verified via the rulesets API), and every weekly scheduled apply run since stage-4 graduation has completed cleanly. The three writes-without-a-PR gates are unchanged and are what make an unattended write acceptable: the write is additive and idempotent with a live pre-write check, scope-minimised to the single Copilot rule on the default branch (it cannot block merges or restrict access), and reversible through the name-guarded `removeCopilotReviewRuleset` rollback. The run-summary step remains the audit record standing in for the PR. With zero open `code-review-bot` findings today the promotion is forward-looking, exactly like the stage-4 flip: a scheduled run acts only when a new gap appears. Demotion is the usual one-line revert — remove the key from `apply-schedule`. The class remains ineligible for auto-merge by construction (settings executors are excluded from `apply-automerge`), so "kept off the allow-list" wording below is historical in the same way as the go-live wording.

## Context

Every cross-repo write the butler performs today is a pull request. ADR-005's five-gate trust model is built on that assumption: gate 4 caps the number of *PRs* per run, dedup keys off an existing-PR check, and the implicit safety net under all of it is that a PR carries a reviewable diff a human can read before it merges and can revert by closing or reverting the merge. "Open a PR a human reviews" is the unit of change.

Standardising GitHub Copilot code review across the portfolio breaks that assumption. Copilot automatic review is not a committed file — it is a `copilot_code_review` rule inside a repository ruleset, created through repository settings or the `POST /repos/{owner}/{repo}/rulesets` REST endpoint (the rule type is supported by the API, go-github, and the Terraform GitHub provider). There is no file to template into a PR, so propagating it the way the butler propagates `code-scanning` or `dependabot-auto-merge` is impossible. The only way to enable it programmatically is a settings-level write: a live API call that changes the repository's configuration the instant it returns.

A settings write has none of a PR's safety affordances. There is no diff to review before it takes effect, no thread to comment on, and reverting means issuing a second API call to delete the ruleset rather than closing an unmerged branch. The whole point of ADR-005 was that cross-repo writes are dangerous because a single fault fans out across the portfolio; a PR-less write removes the one mechanism — pre-merge human review — that bounded the damage of a write that slipped through the other gates. The detection side of the `code-review-bot` standard is therefore safe to ship immediately (it only reads rulesets, and shipped 2026-06-13 routing to the `manual` executor), but the remediation side needs a trust model of its own before the butler is allowed to mutate settings unattended.

## Decision

Permit settings-level (PR-less) writes only for the bounded, additive case of enabling a `copilot_code_review` ruleset, and only under all five ADR-005 gates plus three writes-without-a-PR gates that stand in for the missing pre-merge review.

The five ADR-005 gates carry over unchanged. The settings-apply path is `workflow_dispatch`-only and dry-run fail-closed (any input other than the literal `'false'` only logs the ruleset it would create); it refuses to run unless `require_approval` is `true`; it is bounded by the same per-run cap so a fault touches at most a handful of repos before the batch limit halts it; and every repo name is validated against `^[a-zA-Z0-9._-]+$` before it is interpolated into an API path. Promotion onto the scheduled (no-human) path reuses the per-class `apply-schedule` allow-list from ADR-005's 2026-06-06 amendment, default-closed and reversible. v1 ships with `code-review-bot` **absent** from that allow-list: it runs on the manual `apply.yml` dispatch only, exactly as `codeowners` and `security-md` first shipped. Scheduled promotion is a later, separately-reviewed config change and is held until the existing `codeowners`/`security-md` apply PRs have produced the track record that ADR-007's stage-4 live-graduation gate requires — this ADR does not authorise wiring the class straight onto the scheduled path.

In the executor model (ADR-007), the `code-review-bot` standard routes to a new `executor: 'settings'` value alongside `template`/`agent`/`manual`. The existing `applyGovernanceFindings` PR path filters on `executor === 'template'`, so `settings` findings never leak into it; they are handled by a sibling apply function (`applyCopilotReviewRulesets`) that rides the same gate stack, mirroring how `nudgeStaleDependabotPRs` is a separate action behind the same gates rather than a branch inside the PR path.

Three additional gates replace what the absent PR used to provide.

The first is additive-and-idempotent writes. The butler creates one new, distinctively named ruleset (`repo-butler/copilot-code-review`) carrying only the `copilot_code_review` rule. It never edits or deletes a ruleset it did not create — every mutating call is guarded by an exact match on that name, so a maintainer's hand-tuned rulesets are untouchable by construction. The idempotency guard runs **live at apply time**, not from the OBSERVE-phase snapshot: immediately before writing to a repo, the apply path re-queries the repo's rulesets and skips the write if an active ruleset already requests Copilot review. This closes the stale-snapshot window where a ruleset enabled (by hand or a prior run) between OBSERVE and APPLY would otherwise be duplicated. The ruleset list is paginated (`gh.paginate`, not a single capped request) so the butler cannot miss its own ruleset on a repo with many rulesets and write a duplicate. Detection reads repo-level rulesets only; an org-level ruleset pushed down to a repo is not surfaced, so on org-owned repos a redundant repo-level ruleset could be created — benign (a second active Copilot rule changes nothing) and out of scope here because the portfolio is personally owned, where org rulesets do not apply.

The second is scope minimisation. The created ruleset targets only the default branch, sets enforcement to active, and contains nothing but the single Copilot rule. The exact payload is fixed:

```json
{
  "name": "repo-butler/copilot-code-review",
  "target": "branch",
  "enforcement": "active",
  "conditions": { "ref_name": { "include": ["~DEFAULT_BRANCH"], "exclude": [] } },
  "rules": [
    { "type": "copilot_code_review",
      "parameters": { "review_on_push": true, "review_draft_pull_requests": false } }
  ]
}
```

It deliberately carries no branch-protection, required-status-check, or push restriction, so the worst case of a wrongly-targeted ruleset is benign: Copilot gets asked to review pull requests. It cannot block a merge, lock a branch, or restrict a contributor — the failure modes that make settings writes frightening are excluded by what the ruleset is allowed to contain.

The third is an audit-and-revert record that stands in for the PR. Because there is no PR to point at afterwards, every run — dry-run and live — logs the exact repos mutated and the exact ruleset payload to the run summary, restoring the "see the change before it lands" moment a PR diff provides. Reversibility is a first-class, tested affordance, not a manual afterthought: the butler ships `findButlerCopilotRuleset(gh, owner, repo)` and `removeCopilotReviewRuleset(gh, owner, repo)`, the latter refusing to delete any ruleset whose name is not exactly `repo-butler/copilot-code-review`, so a bad batch can be rolled back across the portfolio through the same tested code path that created it. (A `DELETE /repos/{owner}/{repo}/rulesets/{id}` by hand remains the manual fallback.)

### Token scope (go-live prerequisite, and a gate in its own right)

Creating a repository ruleset requires the GitHub App permission `administration: write` — strictly broader than the `contents: write` + `pull-requests: write` the existing apply path uses, and broad enough to include repository deletion. This is **not** the same token capability as the PR path, and the App installation must be re-permissioned (and re-approved on each target repo) before any live write succeeds; without it the write fails closed with a 403. Granting `administration: write` materially widens the App's blast radius and is therefore treated as an explicit go-live gate: the maintainer makes that grant deliberately, and until then the feature is inert. The code is written to run end-to-end in dry-run with the existing token (it only reads rulesets and logs the payload it would POST), so it is shippable and testable now and goes live only after the scope is granted and dry-run is flipped on a manual dispatch.

A licence note, not a gate: the ruleset only causes reviews to be requested on accounts holding a Copilot plan (Pro/Pro+/Max, or Business/Enterprise), and on public repositories Copilot review consumes no Actions minutes. The butler cannot verify the plan from the API; on a repo whose owner lacks one the ruleset is inert rather than harmful — it requests a reviewer that never arrives — which further bounds the blast radius.

## Consequences

The butler gains a second write modality — settings, not just files — without weakening the portfolio-scale trust model, because the new modality inherits all five existing gates and adds three that specifically answer "what protects us when there is no PR to review?": the write is additive, idempotent (checked live), scope-minimised to a single benign rule, fully audited, and reversible through a tested name-guarded path. The one genuine expansion is the `administration: write` token grant, which is held behind an explicit, deliberate maintainer step rather than assumed.

The cost is that this carves a narrow, explicitly-bounded exception. Future settings-write finding classes (branch protection, required checks, repo flags) are in scope for the ADR-009 trust model in principle, but each needs its own governance-standard definition and its own promotion review, because they carry the merge-blocking and access-restricting failure modes this ADR's benign-worst-case argument deliberately excludes. The moment a settings write can block merges or restrict access, that argument no longer holds and the gate analysis must be redone.

Until the `administration: write` grant is made and dry-run is flipped on a manual dispatch, `code-review-bot` findings are still effectively surfaced for the maintainer to act on: the apply path previews exactly what it would create, and the maintainer can enable the ruleset (or let the previewed write proceed) deliberately.
